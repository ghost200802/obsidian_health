# Ingest 工作流

## 定义

Ingest 工作流是 [[LLM-Wiki知识管理]] 中 Agent 处理原始资料的标准流程。它不只是生成摘要，而是一个包含阅读理解、来源页生成、概念提取、交叉链接和索引更新的完整知识编译过程。

## 流程步骤

### 1. 阅读并理解
- 仔细阅读 [[Raw层]] 中的源文件
- 提取核心观点、关键概念、重要论据
- 识别与已有知识页的潜在关联

### 2. 生成来源页
- 在 /source 下创建 `source-*.md` 文件
- 包含：元信息、核心摘要、关键观点列表、保留价值、关联知识页

### 3. 提取概念
- 从资料中识别值得独立记录的概念
- 已有概念页 → 追加更新，标注来源
- 新概念 → 创建新页面，包含定义、要点、关联，要求符合[[知识点创建工作流]]

### 4. 添加交叉链接
- 使用 Obsidian `[[]]` 双链语法
- 每个页面至少链接 2-3 个其他页面
- 链接应有意义，不是随意堆砌

### 5. 更新索引页
- 某领域页面增多时，创建或更新索引页
- 索引页提供该领域的导航入口

### 6. 验证并提交
- 检查本次新增或修改的 `source/`、`wiki/` 文件，确认 frontmatter、双链、结构和内容无明显问题
- 如发现问题，先修复，再重新检查
- 确认无问题后，执行一次 git 提交和 push

## 关键原则

- **先跑通链路，再优化规模**：第一次只 ingest 一篇资料
- **任务要具体**：不要说"帮我优化知识库"，要说"请 ingest raw/ 中的 xxx 文件"
- **检查产出**：不只看 Agent 回复的文字，要回 [[Obsidian知识库|Obsidian]] 检查实际文件变动
- **收尾闭环**：完成后不仅要检查文件，还要在确认无问题后提交并 push

## 并行执行约束

当多个 Agent 并行 ingest 不同资料时，必须限制共享页面写入范围。

- Agent 可以直接修改：
  - 当前资料对应的 `source/source-*.md`
  - 当前资料新建的专属 `wiki/` 页面
- Agent 不应直接并行修改：
  - 已存在的通用知识点页
  - 索引页
  - 高复用总纲页
  - 其他 Agent 已在处理的共享页面

如果当前资料确实需要补充这些共享页面：

1. 先在本资料来源页中记录相关结论和来源
2. 必要时先落到本资料专属页面
3. 再由主 Agent 或串行合并阶段统一写回共享页面

如果该资料来自已缓存的 PDF：

1. ingest 阶段优先引用 `.derived/pdf_extract/` 下已有的 `.md` / `.json`
2. 单本 ingest 完成后不清理对应 PDF 导出结果
3. 后续主 Agent 串行补充共享知识点页时，可以继续回看同一份导出结果

## 常见错误

- 只做摘要，不提取概念和建立链接 → 变成自动摘要器
- 跳过交叉链接步骤 → 生成孤立页面
- 一次 ingest 太多资料 → 难以验证链路是否通畅
- 多个 Agent 同时改同一知识点页 → 容易产生文本冲突和结构漂移
