# LLM Wiki vs RAG

## 核心区别

| 维度 | [[LLM-Wiki知识管理]] | [[检索增强生成]] |
|------|-------------|---------|
| **核心理念** | 编译知识，持续累积 | 按需检索，即时拼凑 |
| **处理方式** | Agent 主动整理和编译 | 被动切片 + 向量检索 |
| **知识状态** | 沉淀为结构化的中间知识层 | 保持原始切片状态 |
| **回答方式** | 优先站在已整理的 wiki 层回答 | 从原始文档中检索拼凑 |
| **知识增长** | 越用越好用（复利效应） | 不累积，每次重新检索 |
| **维护成本** | Agent 自动维护 + 人引导 | 向量库更新 + embedding 维护 |
| **适用场景** | 长期知识积累、研究、内容创作 | 一次性问答、文档查询 |

## 详细对比

### 普通 RAG 流程
```
文件存入 → 切片 → Embedding → 需要时检索 → 现场拼答案
```
- 优点：实现简单，回答速度快
- 缺点：知识不会累积，每次都从原始数据重新拼凑，无法形成深层理解

### LLM Wiki 流程
```
资料进 raw → Agent 做整理和知识编译 → wiki 层沉淀中间知识 → 提问时优先查这层
```
- 优点：知识持续复利，回答质量越来越高
- 缺点：前期需要投入时间搭建和验证链路

## 本质差异

两者的差别不是能不能回答问题，而是：**你的知识，会不会越用越好用**。

- RAG 解决的是"能不能找到"的问题
- [[LLM-Wiki知识管理]] 解决的是"知识会不会累积"的问题

## 何时选择哪个

- **选 RAG**：需要快速搭建问答系统、文档量巨大但知识不需要深层理解
- **选 LLM Wiki**：做长期知识积累、研究、内容创作，需要知识持续进化
