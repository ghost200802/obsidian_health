# Raw 全量转换与长期派生层工作流

## 定义

Raw 全量转换，指的是先把 `raw/` 中的原始文件批量转换成可长期保留、可复用、可断点续跑的“派生层”，再进入 [[Ingest工作流]] 和 [[知识编译]]。

这里要严格区分三层：

- `raw/` 是只读输入层，不修改原文
- `.derived/` 是长期保留的派生层，保存抽取全文、结构化结果和异常记录
- `source/` 是结构化来源页，不承接大段原始抽取全文

换句话说，派生层不是最终知识页，也不是 `source/` 的替代品；它是 [[Raw层]] 和 [[Wiki层]] 之间的长期基础设施。

## 目标

- 把 `raw/` 中的原始文件先变成可检索、可引用、可复核的全文层
- 避免后续每次更新 `source/` 或 `wiki/` 时重复 OCR 或重复抽取
- 让多轮 Agent、多人协作和串行补充都能共享同一份稳定派生结果
- 为后续 [[串行Ingest队列工作流]] 提供统一的上游输入

## 输出层设计

### 基本原则

- 派生结果统一写入 `.derived/`
- 目录结构尽量镜像 `raw/` 的相对路径
- 一个原始文件对应一个独立派生目录
- 派生层长期保留，不作为一次性临时缓存处理
- `source/` 只保留结构化整理，不直接塞入整篇原文抽取内容

### 目录约定

- PDF：`.derived/pdf_extract/<raw相对路径去扩展名>/`
- DOCX：`.derived/doc_extract/<raw相对路径去扩展名>/`
- DOC：`.derived/doc_extract/<raw相对路径去扩展名>/`
- EPUB：`.derived/epub_extract/<raw相对路径去扩展名>/`
- JPG/PNG：`.derived/image_extract/<raw相对路径去扩展名>/`
- 异常记录：`.derived/conversion_exceptions/<raw相对路径去扩展名>.md`

### 标准产物

每个派生目录至少包含：

- `manifest.json`：记录原始路径、原文件大小、修改时间、转换工具、转换时间、状态
- `full.md`：适合人读和后续交给 Agent 的全文 Markdown
- `full.json`：保留页码、块级结构、识别方式、统计信息
- `assets/`：版面图、截图、中间资源，可选

### PDF 兼容规则

当前仓库已有 [[PDF本地解析工具链]] 和 [[串行Ingest队列工作流]]，它们默认识别：

- `full.auto.md`
- `full.auto.json`

因此 PDF 批量转换阶段仍然优先兼容这套命名，不在本轮工作流里直接破坏现有脚本约定。

## 工具策略

### 第一优先级：MinerU

适用格式：

- `.pdf`
- `.docx`
- `.jpg`

推荐命令形态：

```powershell
.\.venv-mineru\Scripts\mineru.exe -p "<原始文件>" -o "<输出目录>" -b pipeline -m auto -l ch
```

适用场景：

- 版面复杂的 PDF
- 混合文本层与扫描层的 PDF
- 需要保留结构化输出的课程资料、指南、报告和图片类文件

### 第二优先级：现有 PDF 派生脚本

适用格式：

- 已纳入现有队列机制的 PDF

推荐命令形态：

```powershell
python scripts/pdf_extract.py cache "<raw文件>"
```

适用场景：

- 需要继续兼容现有 `.derived/pdf_extract/` 结构
- 需要和现有 `ingest_queue.py` 直接对接

### 回退策略

- `.doc`：先转换为 `.docx`，再交给 MinerU
- `.epub`：单独走 EPUB 提取链，不和 PDF 队列混跑
- 无法自动转换的文件：创建异常记录页，不直接跳过

## 分阶段执行计划

### 阶段 1：先完成 PDF 主体派生层

当前 `raw/` 中共有 `184` 个文件，其中：

- `163` 个 PDF
- `13` 个 DOCX
- `5` 个 DOC
- `2` 个 EPUB
- `1` 个 JPG

由于 PDF 占绝大多数，且现有脚本和队列已经围绕 PDF 建好，所以第一阶段先把 PDF 全部落到长期派生层。

### 阶段 2：补齐非 PDF 派生层

在 PDF 阶段稳定后，再处理：

- DOCX
- DOC
- JPG
- EPUB

这一阶段仍写入 `.derived/`，但不与 PDF 队列混跑，避免批次规则混乱。

### 阶段 3：派生层对接来源页

当某一批派生结果稳定后，再进入：

- 更新对应 `source/source-*.md`
- 补写或增强相关 `wiki/` 页面
- 通过 [[Ingest工作流]] 回写知识层

注意：全量转换阶段的完成，不等于来源页和 wiki 已完成。

## 单批执行步骤

### 1. 盘点当前目录

- 统计文件数与格式分布
- 确定本批只处理一个目录
- 先处理支持度最高的格式

### 2. 执行批量转换

- PDF 优先写入 `.derived/pdf_extract/`
- DOCX、DOC、JPG、EPUB 写入各自格式目录
- 同一目录内允许串行跑多个文件
- 不跨目录混跑，避免状态混乱

### 3. 记录状态

每个文件转换后都要落地状态：

- `success`
- `partial`
- `failed`
- `skipped`

状态统一写入 `manifest.json`；失败或异常另写异常说明页。

### 4. 转换后验证

至少检查：

- 输出目录是否生成
- Markdown 是否可读
- JSON 是否存在
- 原始文件路径是否记录正确
- 文件名、目录名是否可追溯回原始文件

### 5. 进入来源页阶段

转换结果确认可用后，后续才进入：

- 来源页补齐
- 概念提取
- 交叉链接
- wiki 沉淀

## 完成定义

某个原始文件只有在同时满足以下条件时，才算完成“全量转换”：

- 已生成派生目录
- 已生成 Markdown 抽取结果
- 已生成 JSON 或等价结构化结果
- `manifest.json` 已记录状态
- 如遇异常，已在异常记录中说明原因

某个目录只有在目录内所有文件都满足上述条件后，才算完成一批。

## 当前实施边界

当前仓库已经具备：

- PDF 长期派生层规则
- MinerU 本地环境
- PDF 队列脚本与来源页对接思路

当前还需要单独补强的部分：

- DOC 到 DOCX 的稳定转换链
- EPUB 到 Markdown 的稳定提取链
- MinerU 输出目录与仓库镜像目录的统一包装脚本

因此，最稳妥的推进方式是：

1. 先把 PDF 主体跑通
2. 再补齐非 PDF 批量包装
3. 最后把整套流程收敛成统一命令
