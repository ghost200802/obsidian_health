# Raw 全量转换与长期派生层工作流

## 定义

Raw 全量转换，指的是把 `raw/` 中的原始文件批量转换成可长期保留、可复用、可断点续跑的“派生层”。

这里要严格区分三层：

- `raw/` 是只读输入层，不修改原文
- `.derived/` 是长期保留的派生层，保存抽取全文、结构化结果和异常记录
- `source/` 是后续整理层，不属于本工作流的产物

换句话说，派生层不是最终知识页，也不是 `source/` 的替代品；它是 [[Raw层]] 和后续整理工作流之间的长期基础设施。

## 目标

- 把 `raw/` 中的原始文件先变成可检索、可引用、可复核的全文层
- 避免后续每次整理时重复 OCR 或重复抽取
- 让多轮 Agent、多人协作和串行补充都能共享同一份稳定派生结果
- 为后续整理流程提供统一的上游输入

## 工作原则

### 1. 先生成排程，再按排程执行

- 本工作流只定义规则，不直接承载具体批次顺序
- 正式执行前，应先生成或更新一份独立排程
- 排程负责记录批次边界、目录顺序、文件类型分布和当前状态
- 转换执行应以排程为准，而不是临时逐目录即兴推进
- 当前排程统一维护在 [[Raw全量转换排程-进行中]]

### 2. 全量转换不等于所有文件都成功抽取

- “全量转换”指的是 `raw/` 中每个文件都得到一个明确结果
- 这个结果可以是：
  - 成功生成 `.derived/` 产物
  - 暂不支持但已生成异常记录
- 因此，本工作流的完成标准是“所有文件都有可追溯结果”，而不是“所有格式都已成功抽取”

### 3. 按支持度推进，不把未完成链路强行混入主流程

- 当前可直接成功转换的文件类型，应优先批量推进
- 当前还没有稳定成功链路的类型，不应阻塞主流程
- 对这类文件，先在 `.derived/conversion_exceptions/` 中留痕，后续再由专门工作流补齐

## 输出层设计

### 基本原则

- 派生结果统一写入 `.derived/`
- 目录结构尽量镜像 `raw/` 的相对路径
- 一个原始文件对应一个独立派生目录
- 派生层长期保留，不作为一次性临时缓存处理
- `source/` 只保留结构化整理，不直接塞入整篇原文抽取内容

### 目录约定

- PDF：`.derived/pdf_extract/<raw相对路径去扩展名>/`
- DOCX：`.derived/doc_extract/<raw相对路径去扩展名>/`
- DOC：当前优先生成异常记录，待稳定转换链完成后再进入正式派生目录
- EPUB：`.derived/epub_extract/<raw相对路径去扩展名>/`
- JPG/PNG：`.derived/image_extract/<raw相对路径去扩展名>/`
- 异常记录：`.derived/conversion_exceptions/<raw相对路径去扩展名>.md`

### 标准产物

每个派生目录至少包含：

- `manifest.json`：记录原始路径、原文件大小、修改时间、转换工具、转换时间、状态
- `full.md`：适合人读和后续整理使用的全文 Markdown
- `full.json`：保留页码、块级结构、识别方式、统计信息
- `assets/`：版面图、截图、中间资源，可选

### PDF 兼容规则

当前仓库已有 [[PDF本地解析工具链]] 和相关 PDF 队列脚本，它们默认识别：

- `full.auto.md`
- `full.auto.json`

因此 PDF 批量转换阶段仍然优先兼容这套命名，不在本轮工作流里直接破坏现有脚本约定。

## 工具策略

### 第一优先级：MinerU

适用格式：

- `.pdf`
- `.docx`
- `.jpg`

推荐命令形态：

```powershell
.\.venv-mineru\Scripts\mineru.exe -p "<原始文件>" -o "<输出目录>" -b pipeline -m auto -l ch
```

适用场景：

- 版面复杂的 PDF
- 混合文本层与扫描层的 PDF
- 需要保留结构化输出的课程资料、指南、报告和图片类文件

### 第二优先级：现有 PDF 派生脚本

适用格式：

- 已纳入现有队列机制的 PDF

推荐命令形态：

```powershell
python scripts/pdf_extract.py cache "<raw文件>"
```

适用场景：

- 需要继续兼容现有 `.derived/pdf_extract/` 结构
- 需要和现有 PDF 派生脚本直接对接
- 需要优先采用纯本地、低调用成本的长任务运行方式

### 回退策略

- `.doc`：先转换为 `.docx`，再交给 MinerU
- `.epub`：单独走 EPUB 提取链，不和 PDF 队列混跑
- 无法自动转换的文件：创建异常记录页，不直接跳过

### 当前主流程适用边界

- 可直接进入主流程的格式：`.pdf`、`.docx`、`.jpg`、`.jpeg`、`.png`、`.epub`
- 当前尚未补齐稳定成功链路的格式：`.doc`
- 对后者，当前阶段的标准做法是先留异常记录，不与主流程成败混淆

## 分阶段执行计划

### 阶段 0：先生成或刷新排程

- 先盘点目录、格式、数量和优先级
- 把执行顺序、批次边界和例外类型写入排程
- 确认排程后，才启动本地长任务

### 阶段 1：先完成 PDF 主体派生层

由于 PDF 通常是最主要、也是现有工具链最成熟的输入类型，所以第一阶段先把 PDF 全部落到长期派生层。

### 阶段 2：补齐非 PDF 派生层

在 PDF 阶段稳定后，再处理：

- DOCX
- DOC
- JPG
- EPUB

这一阶段仍写入 `.derived/`，但不与 PDF 队列混跑，避免批次规则混乱。

### 阶段 3：结束于派生层校验

当某一批派生结果稳定后，本工作流就在 `.derived/` 层结束。

后续如果需要进入 `source/`、`wiki/` 或知识编译，应该切换到独立的下一步工作流，而不是继续沿用本工作流。

## 批次顺序

具体目录、文件数量、优先级和当前执行顺序，不写在本工作流中，统一维护在：

- [[Raw全量转换排程-进行中]]

## 单批执行步骤

### 1. 先更新排程

- 确认本批目录已经写入排程
- 确认本批状态、优先级和前置条件
- 如果目录结构或文件类型发生变化，先更新排程再继续

### 2. 盘点当前目录

- 统计文件数与格式分布
- 确定本批只处理一个目录
- 识别“可直接成功转换”和“当前只留异常记录”的文件类型
- 先处理支持度最高的格式

### 3. 执行批量转换

- PDF 优先写入 `.derived/pdf_extract/`
- DOCX、DOC、JPG、EPUB 写入各自格式目录
- 同一目录内允许串行跑多个文件
- 不跨目录混跑，避免状态混乱
- 不建议一次性把整个 `raw/` 作为单个超大任务启动
- 更推荐按排程中的目录或批次逐批启动

### 3.1 本地长任务运行原则

- 长文件和整批目录优先交给本地后台任务持续运行，不要求 Agent 持续盯跑
- 推荐入口：`scripts/start_local_derive_job.ps1`
- 任务启动后，本地进程自行持续执行，Agent 不做高频轮询
- 只在任务报错、状态异常、或用户明确要求时，才读取一次状态文件并分析
- 如果某个单文档任务启动后 `10` 分钟仍未完成，可改为每 `10` 分钟做一次轻量进度检查
- 这类检查只用于确认任务仍在正常推进，而不是卡死或无响应
- 不依赖定时快照刷日志，优先使用“事件驱动”的小型进度文件

### 3.2 进度记录原则

- 进度统一写入 `.derived/job_runs/<job_id>/progress.json`
- `progress.json` 始终只保留最新状态，不滚动堆积
- PDF 默认每处理 `10` 页更新一次进度
- 任务结束时再写一次最终状态
- 进度字段至少包括：
  - 时间戳
  - 总文件数与已完成文件数
  - 当前文件序号与当前文件路径
  - 总页数、已处理页数、当前页

### 3.3 Agent 分析边界

- 本地程序持续运行本身不消耗 Agent 调用
- 真正增加调用成本的是 Agent 反复读取日志、解释状态、追问细节
- 因此批量运行时默认采用“本地高频进度，Agent 低频分析”策略
- 对超过 `10` 分钟的单文档任务，允许按 `10` 分钟间隔读取一次 `progress.json`
- 如果进度正常推进，Agent 不应频繁读取大日志
- 如果需要人工检查，优先读取 `progress.json` 这类小文件，而不是整份 `stdout.log`

### 4. 记录状态

每个文件转换后都要落地状态：

- `success`
- `partial`
- `failed`
- `skipped`

状态统一写入 `manifest.json`；失败或异常另写异常说明页。

### 5. 转换后验证

至少检查：

- 输出目录是否生成
- Markdown 是否可读
- JSON 是否存在
- 原始文件路径是否记录正确
- 文件名、目录名是否可追溯回原始文件
- 对暂不支持格式，异常记录是否已经生成且原因明确

### 6. 结束于派生层

转换结果确认可用后，本工作流即结束，不继续要求：

- 来源页补齐
- 概念提取
- 双链回写
- wiki 沉淀

这些属于后续独立工作流。

## 完成定义

某个原始文件只有在同时满足以下条件时，才算完成“全量转换”：

- 已生成派生目录，或已生成异常记录
- 如果属于支持格式，已生成 Markdown 抽取结果
- 如果属于支持格式，已生成 JSON 或等价结构化结果
- 如果属于支持格式，`manifest.json` 已记录状态
- 如遇异常，已在异常记录中说明原因
- 结果已经过最小校验，确认可读、可追溯、可复用

某个目录只有在目录内所有文件都满足上述条件后，才算完成一批。

某一批完成后，本工作流的收尾动作是确认 `.derived/` 层产物正确，而不是继续推进到 `source/` 或 `wiki/`。

## 当前实施边界

当前仓库已经具备：

- PDF 长期派生层规则
- MinerU 本地环境
- PDF 队列脚本基础
- 本地后台长任务入口与轻量状态检查脚本

当前还需要单独补强的部分：

- DOC 到 DOCX 的稳定转换链
- MinerU 输出目录与仓库镜像目录的统一包装脚本

因此，最稳妥的推进方式是：

1. 先生成并确认排程
2. 先把支持度最高的批次跑通
3. 再补齐非 PDF 的稳定转换链
4. 最后把整套转换流程收敛成统一命令
